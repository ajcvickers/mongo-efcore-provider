# DOM-free materialization (sub-project C) — results

> **OUTCOME: hypothesis disproven; change REVERTED.** The `RawBsonDocument` swap *regressed* native-shape
> allocations by +68–82% (see below) because the random-access by-name shaper re-scans the byte buffer and
> allocates a fresh `BsonValue` per field access on `RawBsonDocument`, whereas a parsed `BsonDocument` caches
> them in a one-time element dictionary. The `src/` change was reverted (the native path is back to
> `BsonDocument`, no regression); this document is retained as the measured evidence. The real materialization
> win requires a forward-only streaming-reader shaper (read each field once, zero per-field `BsonValue`
> allocation) — a separate future sub-project. `RawBsonDocument` is the right transport only *behind* that reader.

## Reading & recommendation

- **This is an allocation REGRESSION, not the expected reduction.** Every shape that flows RawBsonDocument
  through the shaper (the four native shapes) allocates **~70–82% MORE** than the A baseline. The one shape
  that falls back to driver-LINQ (`Projection_ToList`, trailing user `Select`) is byte-identical to baseline
  (8.27 MB) — exactly the control that isolates the change to the RawBsonDocument path.
- **Root cause: by-name field access on RawBsonDocument re-materializes per call.** The shaper reads fields via
  `BsonBinding`, which calls `document.TryGetValue(name)` / `parent.GetValue(name)`. On a parsed `BsonDocument`
  these hit a one-time element dictionary and return cached `BsonValue` references. On a `RawBsonDocument` they
  are overridden to **scan the byte buffer and allocate a fresh `BsonValue` (and a fresh nested
  RawBsonDocument/BsonDocument slice for the owned `Address`) on every access**. The shaper touches every
  scalar plus the owned sub-document for each of the 10,000 rows, so RawBsonDocument trades the one-time DOM
  build for repeated per-field re-materialization — net more allocation, and slower (the +36–58% Mean). Eager
  `RawBsonDocument` → `BsonValue` materialization per accessed field, plus EF change-tracking / identity /
  snapshot allocations, are all still present and unchanged; RawBsonDocument removed the element dictionary but
  added strictly more `BsonValue` churn on top of everything else.
- **Recommendation: do NOT ship RawBsonDocument as the materialization path as-is.** It regresses the headline
  metric it was meant to improve. The correct lever is the **full streaming-reader shaper sub-project**:
  rewrite `BsonBinding` + shaper codegen to read each row **once, forward-only** from an `IBsonReader` over the
  raw bytes, writing directly into the CLR shape with **zero per-field `BsonValue` allocation** and no element
  dictionary. RawBsonDocument is the right *transport* (lazy, byte-backed, disposable) but only pays off behind
  a forward-only reader that visits each field exactly once — not behind the current random-access by-name
  shaper, which is pathological on it. The residual gap to the spike's ~65% pure-materialization win is
  therefore not "partial savings left on the table" — under the current shaper RawBsonDocument moves the metric
  the wrong way, so the full streaming-reader shaper is **required**, not merely a nice-to-have follow-up. As an
  interim, either revert the native path to parsed `BsonDocument` (no regression, no win) or gate RawBsonDocument
  off until the streaming shaper lands.
