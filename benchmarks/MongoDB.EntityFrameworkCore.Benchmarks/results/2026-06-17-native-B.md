# Native MQL (sub-project B) — validation results

**Run on:** 2026-06-17, Apple M4 Max (1 CPU, 14 logical / 14 physical cores) / macOS Tahoe 26.5.1 (25F80) [Darwin 25.5.0] / .NET SDK 10.0.301 / .NET 10.0.9, Arm64 RyuJIT.
**EF config:** EF10 / net10.0. **MongoDB:** replica set at `mongodb://localhost:27017`.
**Switch:** `MONGODB_EF_NATIVE_QUERY` (`auto` default / `force` / `off`).

This validates sub-project B (native MQL generation for single-collection `$match`/`$sort`/`$skip`/`$limit`):
(1) no regressions in `auto`, (2) native coverage in `force`, (3) benchmark vs the committed baseline.

### Regressions found and fixed

Before these fixes, `auto` mode had ~95 Query failures (per-assembly, stable count) introduced by sub-project B.
Root causes and fixes (all in `src/`):

1. **Projection routing dropped fallback push-down** (`Query/Visitors/MongoShapedQueryCompilingExpressionVisitor.cs`).
   B disabled push-down for *all* projections whenever native mode was on, sending scalar/anonymous projections
   that the native pipeline cannot translate through the client-side-shaper path — which then failed with
   `No coercion operator BsonDocument -> Int32` / `cannot be used for return type IEnumerable<String>`.
   Fix: restore push-down unconditionally; engage native translation **only** on the entity / mixed
   client-side-shaper path (new `nativeEligible` flag), never on the identity-shaper push-down path.

2. **Native MQL was never logged** (`Diagnostics/MongoLoggerExtensions.cs`, `Storage/MongoClientWrapper.cs`).
   `ExecutedMqlQuery` read the driver-LINQ provider's `LoggedStages`, which is empty for native queries, so
   every native `AssertMql` baseline saw an empty pipeline (`Orders.`). Fix: log the `NativePipeline` stages
   for native queries, and set the log action **before** issuing the aggregate so MQL is still logged when
   execution throws (matches the driver-LINQ "log even when the driver throws" contract).

3. **Predicate value mistranslation** (`Query/NativeTranslation/MongoPredicateTranslator.cs`).
   `ToBsonValue` cast hard via the property serializer and threw raw `InvalidCastException`
   (e.g. `Int32 -> UInt16`, `Int64 -> Nullable<UInt32>`) instead of falling back. Fix: coerce the value to the
   property CLR type, and map any serialization failure to `NativeTranslationNotSupportedException` (→ fallback).

4. **MQL-shape divergence from the driver** (`MongoPredicateTranslator.cs`, `MongoPipelineTranslator.cs`).
   Native emitted differently-shaped (but equivalent) MQL than the committed baselines:
   - `!boolProp` rendered `{f:false}` instead of `{f:{$ne:true}}` (also a *semantic* difference for missing fields).
   - Chained `AndAlso` / `OrElse` nested redundantly instead of flattening to a single `$and` doc / `$or` array,
     including merging same-field comparisons (`x>a && x<b` → `{x:{$gt:a,$lt:b}}`).
   - Multiple `Where` clauses were merged; the driver emits one `$match` stage per `Where`.
   Fix: match the driver rendering (`$ne:true`, `$and`/`$or` flattening with same-field operator merge,
   one `$match` per `Where`).

5. **Out-of-scope shapes silently mistranslated → now fall back** (`MongoPipelineTranslator.cs`,
   `MongoPredicateTranslator.cs`, `NativeTranslation/NativeExpressionHelpers.cs`). Tightened the translators to
   throw `NativeTranslationNotSupportedException` (→ fallback in auto) instead of producing wrong results, for:
   trailing `Select` projections; `Take(0)`/non-positive limit (driver validates client-side); repeated or
   out-of-order `Skip`/`Take`; filter/sort after paging; a numeric **cast** on the member side of a comparison
   (e.g. `(double)decimalProp > 100.0`); and **composite-primary-key** components (stored nested under `_id`,
   which the native element resolver does not address — was returning zero rows).

No test was weakened, skipped, or disabled.

## 2. Force-mode native coverage

`MONGODB_EF_NATIVE_QUERY=force` makes the native path **throw** instead of falling back, so passing tests are
exactly those the native slice translates end-to-end. (UnitTests do not hit the DB.)

| Assembly | Passed natively | Total (excl. skipped) | Coverage |
|---|---:|---:|---:|
| SpecificationTests | 2331 | 4345 | ~54% |
| FunctionalTests | 310 | 544 | ~57% |

That is the coverage of B's filter/sort/paging-over-whole-entities slice. The ~2,250 force-mode failures are
the expected out-of-scope shapes throwing `NativeTranslationNotSupportedException` (1,344 occurrences) — this is
the coverage signal, **not** regressions.

### Top fallback reasons (most common distinct messages)

| Count | Reason | Category |
|---:|---|---|
| 572 | predicate node not supported (`Call` 277, `Equal` 155, `Not` 44, `NotEqual` 40, `Constant` 20, …) | complex predicates / method calls (`Contains`, `StartsWith`, `string.*`, null-coalesce, etc.) |
| 272 | pipeline operator not supported | `VectorSearch` (112), `LeftJoin` (83), `Distinct` (30), `Union` (18), `Join` (16), `Concat` (10), `OfType` (3) |
| 71 | trailing projection (`Select`) | projections route to driver-LINQ / client-side shaper |
| 24 | cast on member operand | e.g. `(double)decimalProp > x` |
| 22 | `Where` after `Skip`/`Take` | non-canonical paging order |
| 14 | value node not a constant/parameter | computed / non-evaluable comparands |
| 10 | repeated or out-of-order `Skip`/`Take` | e.g. `Take().Skip()`, multiple `Skip` |
| ~30 | unsupported sort key | null-coalesce / parameter / subquery sort keys |

The dominant gaps (method-call predicates, joins, grouping, set ops, scalar terminators, vector search,
projections) are all outside B's slice and correctly defer to the driver-LINQ path.

## 3. Benchmark — native (auto) vs committed baseline

Baseline: `results/2026-06-16-baseline.md` (current driver-LINQ path). Native run is the default `auto` switch,
so 4 of the 5 shapes now translate natively (`Where_ToList`, `OrderBy_Take`, `Tracked_ToList`,
`NoTracking_ToList`); `Projection_ToList` has a trailing `Select` and falls back to driver-LINQ.

| Method | Baseline Mean | Native Mean | Δ Mean | Baseline Alloc | Native Alloc | Δ Alloc | Native path |
|---|---:|---:|---:|---:|---:|---:|:--|
| Where_ToList      |  55.978 ms |  53.618 ms |  −4.2% |  73.46 MB |  72.86 MB | −0.8% | native |
| Projection_ToList |  13.805 ms |  13.078 ms |  −5.3% |   8.27 MB |   8.27 MB |  0.0% | fallback |
| OrderBy_Take      |   3.492 ms |   3.102 ms | −11.2% |   1.55 MB |   1.54 MB | −0.6% | native |
| Tracked_ToList    | 105.559 ms | 106.775 ms |  +1.2% | 147.04 MB | 145.82 MB | −0.8% | native |
| NoTracking_ToList |  70.214 ms |  75.484 ms |  +7.5% | 125.83 MB | 124.92 MB | −0.7% | native |

**Reading.** As expected for B, allocation changes are within noise (≤1%): sub-project B keeps the
BsonDocument-DOM materialization (the shaper is unchanged), so the large materialization-allocation win is
deferred to sub-project C. B's effect is on the *translation* path — bypassing the driver-LINQ expression
compiler in favor of a directly-constructed `$match`/`$sort`/`$skip`/`$limit` pipeline. Mean times move within
run-to-run variance (the `±` errors and known InProcess outliers overlap these deltas); `OrderBy_Take` shows the
most favorable movement and the two large tracked/no-tracking enumerations are dominated by materialization, so
they are flat-to-slightly-noisier. Net: **no performance regression, modest translation-overhead improvement,
allocation unchanged** — consistent with B's scope.

> Absolute times are only comparable on the same machine/server; allocations are the portable signal and are
> essentially unchanged, exactly as predicted for a translation-only change.
