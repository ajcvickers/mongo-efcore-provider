### Regressions found and fixed

Before these fixes, `auto` mode had ~95 Query failures (per-assembly, stable count) introduced by sub-project B.
Root causes and fixes (all in `src/`):

1. **Projection routing dropped fallback push-down** (`Query/Visitors/MongoShapedQueryCompilingExpressionVisitor.cs`).
   B disabled push-down for *all* projections whenever native mode was on, sending scalar/anonymous projections
   that the native pipeline cannot translate through the client-side-shaper path ‚Äî which then failed with
   `No coercion operator BsonDocument -> Int32` / `cannot be used for return type IEnumerable<String>`.
   Fix: restore push-down unconditionally; engage native translation **only** on the entity / mixed
   client-side-shaper path (new `nativeEligible` flag), never on the identity-shaper push-down path.

2. **Native MQL was never logged** (`Diagnostics/MongoLoggerExtensions.cs`, `Storage/MongoClientWrapper.cs`).
   `ExecutedMqlQuery` read the driver-LINQ provider's `LoggedStages`, which is empty for native queries, so
   every native `AssertMql` baseline saw an empty pipeline (`Orders.`). Fix: log the `NativePipeline` stages
   for native queries, and set the log action **before** issuing the aggregate so MQL is still logged when
   execution throws (matches the driver-LINQ "log even when the driver throws" contract).

3. **Predicate value mistranslation** (`Query/NativeTranslation/MongoPredicateTranslator.cs`).
   `ToBsonValue` cast hard via the property serializer and threw raw `InvalidCastException`
   (e.g. `Int32 -> UInt16`, `Int64 -> Nullable<UInt32>`) instead of falling back. Fix: coerce the value to the
   property CLR type, and map any serialization failure to `NativeTranslationNotSupportedException` (‚Üí fallback).

4. **MQL-shape divergence from the driver** (`MongoPredicateTranslator.cs`, `MongoPipelineTranslator.cs`).
   Native emitted differently-shaped (but equivalent) MQL than the committed baselines:
   - `!boolProp` rendered `{f:false}` instead of `{f:{$ne:true}}` (also a *semantic* difference for missing fields).
   - Chained `AndAlso` / `OrElse` nested redundantly instead of flattening to a single `$and` doc / `$or` array,
     including merging same-field comparisons (`x>a && x<b` ‚Üí `{x:{$gt:a,$lt:b}}`).
   - Multiple `Where` clauses were merged; the driver emits one `$match` stage per `Where`.
   Fix: match the driver rendering (`$ne:true`, `$and`/`$or` flattening with same-field operator merge,
   one `$match` per `Where`).

5. **Out-of-scope shapes silently mistranslated ‚Üí now fall back** (`MongoPipelineTranslator.cs`,
   `MongoPredicateTranslator.cs`, `NativeTranslation/NativeExpressionHelpers.cs`). Tightened the translators to
   throw `NativeTranslationNotSupportedException` (‚Üí fallback in auto) instead of producing wrong results, for:
   trailing `Select` projections; `Take(0)`/non-positive limit (driver validates client-side); repeated or
   out-of-order `Skip`/`Take`; filter/sort after paging; a numeric **cast** on the member side of a comparison
   (e.g. `(double)decimalProp > 100.0`); and **composite-primary-key** components (stored nested under `_id`,
   which the native element resolver does not address ‚Äî was returning zero rows).

No test was weakened, skipped, or disabled.

## 4. Review fixes

A code review of the native MQL path raised two correctness items and one coverage question. All three are
resolved below; auto-mode regression suite re-run after the changes is **0 failures** (UnitTests 8/0,
FunctionalTests 544/0 (44 skipped), SpecificationTests 4345/0 (18 skipped) ‚Äî totals **4897 passed / 0 failed /
62 skipped**, identical to the pre-fix baseline in ¬ß1).

### FIX 1 ‚Äî comparison operator now mirrored when the member is on the RIGHT (blocking correctness bug)

`MongoPredicateTranslator.TranslateComparison` mapped `be.NodeType` directly to the `$`-operator regardless of
which operand the property resolved on. For a constant-on-left comparison such as `Where(p => 30 < p.Count)`
that emitted `{ Count: { $lt: 30 } }` ‚Äî i.e. `Count < 30` ‚Äî silently returning the wrong rows in `auto` mode.

Fix: when the member resolves on the **right** operand, mirror the relational operator before mapping
(`<`‚Üî`>`, `<=`‚Üî`>=`; `==`/`!=` are symmetric and unchanged), via a new `Mirror(ExpressionType)` helper and an
"effective" node type (`memberOnLeft ? be.NodeType : Mirror(be.NodeType)`).

Proof (force mode, so the native path is genuinely exercised, not falling back): constant-on-left queries now
return identical row counts to the member-on-left equivalents over a 100-row seed (`Count` = 0..99):

| Query | Expected rows | Native result |
|---|---:|---:|
| `30 < Count`  / `Count > 30`  | 69 | 69 / 69 |
| `30 >= Count` / `Count <= 30` | 31 | 31 / 31 |
| `30 <= Count` (‚áî `Count >= 30`) | 70 | 70 |
| `30 > Count`  (‚áî `Count < 30`)  | 30 | 30 |

Before the fix `30 < Count` returned 30 rows (the `$lt` mistranslation) instead of 69.

### FIX 2 ‚Äî equality/inequality and bare-bool against nullable fields now fall back (latent hazard)

A bare `{ field: value }` (from `==`), `{ field: { $ne: x } }` (from `!=`), `{ field: true }` (bare bool) or
`{ field: { $ne: true } }` (`!bool`) can diverge from the driver-LINQ rendering for documents where the field
is absent/null. Conservatively, `MongoPredicateTranslator` now throws `NativeTranslationNotSupportedException`
(‚Üí driver-LINQ fallback) when the resolved property `IReadOnlyProperty.IsNullable` is true, for:
`==`/`!=` comparisons (in `TranslateComparison`), the bare-bool case, and the `!bool` (`Not`) case.
Non-nullable properties (the common case, e.g. a non-null `bool Active`) keep translating natively. A query
that newly falls back is still correct (the driver path handles it), which the auto-mode suite confirms.

### INVESTIGATE 3 ‚Äî whole-entity filter/sort/paging queries DID fall back; now made native (with the synthetic Select handled)

**Finding (confirmed empirically).** With `MONGODB_EF_NATIVE_QUERY=force`, a bare whole-entity query
(`ctx.Customers.Where(c => c.Active).ToList()`) threw `Native pipeline does not support a trailing projection
(Select)`. EF Core appends a **synthetic trailing `Select`** to entity-returning queries to drive
materialization; the captured chain was:

```
EntityQueryRootExpression.Where(c => c.Active).Select(c => [IncludeExpression])
```

The pipeline translator rejected any trailing `Select`, while only the *mixed-projection* path stripped one
first ‚Äî so entity-returning filter/sort/paging queries (B's headline slice) were falling back to driver-LINQ in
`auto`, contrary to the ¬ß3 benchmark table's "native" labels for `Where_ToList` / `Tracked_ToList` /
`NoTracking_ToList` (those labels were optimistic; the synthetic `Select(IncludeExpression)` made them fall back).

**Decision: made native (safe).** The synthetic `Select`'s lambda body is an `IncludeExpression` ‚Äî pure entity
materialization (owned-type / navigation loading), **not** a user transformation. The native path returns whole
`BsonDocument`s and the client-side entity shaper performs that materialization itself, so the `Select` does not
affect the emitted `$match`/`$sort`/`$skip`/`$limit` pipeline. `MongoPipelineTranslator` now recognizes this
exact shape (`IsEntityMaterializationSelect`: a 2-arg `Queryable.Select` whose selector body is an
`IncludeExpression`) and skips it instead of throwing. Any *other* `Select` (a real server-side projection)
still throws and falls back ‚Äî so genuine projections are unaffected.

This is safe and was implemented:
- Force-mode smoke now translates `Where`, `OrderBy().Take()`, `ToList()` (tracked), and `AsNoTracking().ToList()`
  natively ‚Äî `SMOKE OK: where=50, proj=100, ordered=10, tracked=100, noTrack=100, withCity=100`. The owned
  `Address.City` round-trips for all 100 rows through the native path (`withCity=100`), confirming the
  client-side shaper handles the skipped `IncludeExpression` materialization. The user `Select` projection
  (`proj`) still falls back, as intended.
- Auto-mode regression suite is **0 failures** across all three assemblies (above), including the
  SpecificationTests `AssertMql` golden-path baselines ‚Äî so the now-native entity queries render MQL equivalent
  to the driver-LINQ baselines (no shape divergence).
